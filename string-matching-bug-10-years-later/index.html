Finding a bug in core string matching function... 10 years later

2023-11-18

TLDR; Core string matching function that has been in production for over 10 years turns out to be broken due to compiler shenanigans. Here's an example of the comparison with a single character:

unsigned char char1 = '8';
unsigned char char2 = 'X';
int matches = (char1 == char2) ||
              /* Case insensitive comparison */
              (char1 - 'A' <= (unsigned char)'Z' - 'A' && char1 - 'A' + 'a' == char2) ||
              (char1 - 'a' <= (unsigned char)'z' - 'a' && char1 - 'a' + 'A' == char2);
// matches is 1 0_o

The funny part is that the code worked fine before because it used std::tolower.

If you're interested in game engines or you like seeing me go insane, you might want to keep reading. :)

Setting the stage

Enter: Zombie Master: Reborn, or ZMR for short, is an RTS/FPS hybrid zombie shooter game made on the Source engine. It is based on an older Source engine game called, you guessed it, Zombie Master. One of the goals of ZMR is to keep backwards compatibility with maps made for the original game. ZMR inherits hundreds of user-made maps, often dating back 15 years.

One of these user-made maps is Bman Battle, made by a talented mapper Byblo. It is essentially the game Bomberman, but made for Zombie Master. Each player tries to kill the others by placing bombs on the ground. Last one alive wins and the level advances. It was loved by majority of the community thanks to its unique premise and fun gameplay. A lot of effort went into making it, so it was sad to see it broken in ZMR. I also loved this map, so I was determined to fix it. How deep could the problem go?

Bman Battle by Byblo. It features multiple levels, which was a novel feature at the time of its release.

Bombs being non-solid is no fun; you can't trap players into corners!

Looking Deep Into the Code

At this point I attached my debugger and started looking through the code. Let's start with the basics. Does the trigger_multiple get spawned? By putting a breakpoint into CTriggerMultiple::Spawn, we can ensure that some trigger_multiple gets spawned. Looking into the member variables we can see m_iName variable with value pg11. That confirms it.

void CTriggerMultiple::Spawn( void )
{
	BaseClass::Spawn();

	InitTrigger(); // Hurray, pg11 gets here, so it spawns!
}

We know that the bomb turns solid when output OnEndTouchAll is triggered. Okay, let's look if that gets triggered. We'll have to jump one class down into CBaseTrigger::EndTouch and put a breakpoint there:

	if ( IsTouching( pOther ) )
	{
        m_OnEndTouch.FireOutput(pOther, this); // pg11 never gets here :/

That is very odd. Does this mean the player is not touching it? IsTouching just checks if the entity is in a list. Okay, let's jump into CBaseTrigger::StartTouch which is responsible for adding entities into the list:

void CBaseTrigger::StartTouch(CBaseEntity *pOther)
{
	if (PassesTriggerFilters(pOther) )
	{

I can see the player touching pg11, but it never passes the checks in CBaseTrigger::PassesTriggerFilters. It ends up boiling down into this:

bool CBaseTrigger::PassesTriggerFilters(CBaseEntity *pOther)
{
    // Bunch of other code...

    CBaseFilter *pFilter = m_hFilter.Get();
    return (!pFilter) ? true : pFilter->PassesFilter( this, pOther ); // Returns false 🤔
}

The plot thickens, and I turn mad

We have a filter that is suppose to pass, but doesn't. 🤔

When I looked at the decompiled map, I saw that the trigger had a filter called f_pnX, which has a filter value of pn*. That means in order for an entity to pass the filter, it needs a name starting with "pn". The thing is, the map sets all player names to pn1, pn2, pn3, etc. Meaning the filter should pass. This isn't helping, let's dig further. The PassesFilter method boils down to this:

bool CBaseEntity::NameMatchesComplex( const char *pszNameOrWildcard )
{
    // pszNameOrWildcard: pn8 (Filter value)
    // m_iName: pn11 (Player name)
	return NamesMatch( pszNameOrWildcard, m_iName );
}

Hold on, why is the filter value pn8? It's suppose to be pn*. The code makes sense; pn11 and pn8 do not match. The filter does not pass.

At this point I started questioning if I had done something wrong. Maybe I had followed the wrong entity? Are there multiple triggers with the same name? Maybe the map renames triggers? So many maybes...

The culprit

Let's look at this in its full glory:

FORCEINLINE bool NamesMatch( const char *pszQuery, string_t nameToMatch )
{
	if ( nameToMatch == NULL_STRING )
		return (!pszQuery || *pszQuery == 0 || *pszQuery == '*');

	const char *pszNameToMatch = STRING(nameToMatch);

	// If the pointers are identical, we're identical
	if ( pszNameToMatch == pszQuery )
		return true;

	while ( *pszNameToMatch && *pszQuery )
	{
		unsigned char cName = *pszNameToMatch;
		unsigned char cQuery = *pszQuery;
		// simple ascii case conversion
		if ( cName == cQuery )
			;
		else if ( cName - 'A' <= (unsigned char)'Z' - 'A' && cName - 'A' + 'a' == cQuery )
			;
		else if ( cName - 'a' <= (unsigned char)'z' - 'a' && cName - 'a' + 'A' == cQuery )
			;
		else
			break;
		++pszNameToMatch;
		++pszQuery;
	}

	if ( *pszQuery == 0 && *pszNameToMatch == 0 )
		return true;

	// @TODO (toml 03-18-03): Perhaps support real wildcards. Right now, only thing supported is trailing *
	if ( *pszQuery == '*' )
		return true;

	return false;
}
